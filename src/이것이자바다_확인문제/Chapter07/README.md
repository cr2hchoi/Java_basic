## 1. 자바의 상속 (Inheritance)
**Q. 자바의 상속에 대한 설명 중 틀린 것은 무엇입니까?**
- [x] ➊ 자바는 다중 상속을 허용한다.
- [ ] ➋ 부모의 메소드를 자식 클래스에서 재정의(오버라이딩)할 수 있다.
- [ ] ➌ 부모의 private 접근 제한을 갖는 필드와 메소드는 상속의 대상이 아니다.
- [ ] ➍ final 클래스는 상속할 수 없고, final 메소드는 오버라이딩할 수 없다.

> **정답: ➊**
> **해설**: 자바는 클래스 간의 **다중 상속을 허용하지 않는다.** 즉, `extends` 뒤에는 단 하나의 부모 클래스만 올 수 있다. 다중 상속을 허용할 경우 발생할 수 있는 '다이아몬드 문제(여러 부모가 동일한 이름의 메소드를 가질 때 어떤 것을 상속받을지 모호해지는 문제)'를 방지하기 위함이다. 대신 인터페이스(Interface)를 통해 다중 구현을 지원한다.


### **나머지 보기 설명**
*   **➋ 오버라이딩**: 자식 클래스는 부모로부터 물려받은 메소드가 자신의 용도에 맞지 않을 경우 내용을 다시 쓸 수 있다.
*   **➌ private 멤버**: `private`으로 선언된 필드나 메소드는 해당 클래스 내부에서만 접근 가능하므로, 자식 클래스라도 상속받거나 직접 사용할 수 없다.
*   **➍ final 키워드**: 클래스에 `final`이 붙으면 상속의 종착점이 되어 더 이상 자식을 가질 수 없고, 메소드에 붙으면 자식이 해당 메소드를 고쳐 쓸(오버라이딩) 수 없다.

## 2. 클래스 타입 변환 (Type Conversion)
**Q. 클래스 타입 변환에 대한 설명 중 틀린 것은 무엇입니까?**
- [ ] ➊ 자식 객체는 부모 타입으로 자동 타입 변환된다.
- [x] ➋ 부모 객체는 어떤 자식 타입으로도 강제 타입 변환된다.
- [ ] ➌ 자동 타입 변환을 이용해서 필드와 매개변수의 다형성을 구현한다.
- [ ] ➍ 강제 타입 변환 전에 instanceof 연산자로 변환 가능한지 검사하는 것이 좋다.

> **정답: ➋**
> **해설**: 부모 객체라고 해서 무조건 자식 타입으로 강제 변환될 수 있는 것이 아니다. **자식 타입이 부모 타입으로 자동 변환된 상태에서만** 다시 원래의 자식 타입으로 강제 변환(Casting)이 가능하다. 처음부터 부모 객체로 생성된 인스턴스는 자식 타입으로 변환할 수 없다.


### **주요 개념 정리**

#### **자동 타입 변환 (Promotion)**
*   `부모클래스 변수 = 자식객체;` 
*   자식은 부모의 특징을 모두 가지고 있으므로 별도의 작업 없이 자동으로 변환된다.
*   이 기능을 통해 하나의 부모 타입으로 여러 자식 객체를 관리하는 **다형성**을 구현한다.

#### **강제 타입 변환 (Casting)**
*   `자식클래스 변수 = (자식클래스) 부모타입변수;` 
*   부모 타입에 저장된 자식 객체의 원래 멤버들을 다시 사용하고 싶을 때 사용한다.
*   **주의**: 원래 자식 객체였던 것만 돌아갈 수 있으므로 매우 신중해야 한다.

#### **instanceof 연산자**
*   변수가 참조하고 있는 실제 객체가 어떤 타입인지 확인할 때 사용한다.
*   잘못된 강제 타입 변환으로 인해 발생하는 `ClassCastException` 오류를 방지하기 위한 안전장치이다.

## 3. final 키워드
**Q. final 키워드에 대한 설명으로 틀린 것은 무엇입니까?**

- [x] ➊ final 클래스는 부모 클래스로 사용할 수 있다.
- [ ] ➋ final 필드는 초기화된 후에는 변경할 수 없다.
- [ ] ➌ final 메소드는 재정의(오버라이딩)할 수 없다.
- [ ] ➍ static final 필드는 상수를 말한다.



### **💡 해설**
*   **➊ final 클래스**: 클래스 선언 시 `final` 키워드가 붙으면 이는 '최종적인' 클래스임을 의미한다. 더 이상 상속할 수 없는 클래스가 되므로 **자식 클래스를 만들 수 없으며, 따라서 부모 클래스가 될 수 없다**.
*   **➋ final 필드**: 필드에 `final`이 붙으면 초기값이 저장된 후에는 프로그램 실행 도중에 값을 변경할 수 없다.
*   **➌ final 메소드**: 메소드 선언 시 `final`이 붙으면 자식 클래스에서 이 메소드를 재정의(오버라이딩)하여 내용을 수정할 수 없다.
*   **➍ static final**: 객체마다 존재하지 않고 클래스에만 존재하며(`static`), 값을 변경할 수 없는(`final`) 특성을 가지기 때문에 관례적으로 이를 **상수**라고 부른다.


| 대상 | 효과 | 설명 |
| :--- | :--- | :--- |
| **클래스** | **상속 불가** | 부모 클래스가 될 수 없음 (최종 클래스) |
| **메소드** | **오버라이딩 불가** | 자식 클래스에서 재정의할 수 없음 |
| **필드** | **값 수정 불가** | 초기화 이후 데이터 변경 금지 |

## 4. 메소드 오버라이딩 (Method Overriding)
**Q. 오버라이딩(Overriding)에 대한 설명으로 틀린 것은 무엇입니까?**

- [ ] ➊ 부모 메소드의 시그너처(리턴 타입, 메소드명, 매개변수)와 동일해야 한다.
- [ ] ➋ 부모 메소드보다 좁은 접근 제한자를 붙일 수 없다. (예: 부모 public → 자식 private 불가능)
- [ ] ➌ @Override 어노테이션을 사용하면 재정의가 확실한지 컴파일러가 검증한다.
- [x] ➍ protected 접근 제한을 갖는 메소드는 다른 패키지의 자식 클래스에서 재정의할 수 없다.

---

### **💡 해설**
*   **➊ 메소드 시그너처**: 오버라이딩은 부모가 가진 기능을 자식이 동일한 모양으로 다시 쓰는 것이므로, 이름·매개변수·리턴 타입이 모두 같아야 한다.
*   **➋ 접근 제한자**: 자식은 부모의 접근 범위를 줄일 수 없다. 부모가 `public`인데 자식이 `private`으로 숨기는 것은 불가능하다. 반대로 범위를 넓히는 것(부모 `default` → 자식 `public`)은 가능하다.
*   **➌ @Override**: 이 어노테이션은 컴파일러에게 "이건 오버라이딩한 거야"라고 알려주는 신호이다. 만약 부모에게 없는 메소드이거나 철자가 틀리면 컴파일 에러를 발생시켜 실수를 방지한다.
*   **➍ protected 접근 제한**: `protected`는 **같은 패키지** 혹은 **다른 패키지에 있더라도 상속 관계인 자식 클래스**라면 접근을 허용한다. 따라서 다른 패키지의 자식 클래스에서도 얼마든지 오버라이딩할 수 있다.

---

### **📌 접근제한자 모음 ** 
| 접근 제한자 | 오버라이딩 시 자식 클래스에서의 범위 변경 |
| :--- | :--- |
| **부모 public** | 자식은 반드시 **public**만 가능 |
| **부모 protected** | 자식은 **protected** 또는 **public** 가능 |
| **부모 default** | 자식은 **default, protected, public** 가능 |
| **부모 private** | 오버라이딩 자체가 **불가능** (상속 대상 아님) |

## 5. 추상 클래스 (Abstract Class)
**Q. 추상 클래스에 대한 설명으로 틀린 것은 무엇입니까?**

- [ ] ➊ 직접 객체를 생성할 수 없고, 상속만 할 수 있다.
- [x] ➋ 추상 메소드를 반드시 가져야 한다.
- [ ] ➌ 추상 메소드는 자식 클래스에서 재정의(오버라이딩)할 수 있다.
- [ ] ➍ 추상 메소드를 재정의하지 않으면 자식 클래스도 추상 클래스가 되어야 한다.


### **💡 해설**
*   **➊ 객체 생성 불가**: 추상 클래스는 실체 클래스의 공통적인 특성을 추출해 놓은 '미완성 설계도'와 같다. 따라서 `new` 연산자를 사용하여 직접 객체를 생성할 수 없으며, 반드시 자식 클래스로 상속해서 사용해야 한다.
*   **➋ 추상 메소드 보유 여부**: 추상 클래스라고 해서 **반드시 추상 메소드를 가져야 하는 것은 아니다**. 추상 메소드가 하나도 없더라도 클래스 선언에 `abstract` 키워드를 붙이면 추상 클래스가 된다. (반대로, 추상 메소드가 하나라도 있다면 해당 클래스는 반드시 추상 클래스여야 한다.)
*   **➌ 추상 메소드 재정의**: 추상 메소드는 부모 클래스에서 선언부만 있고 실행 내용이 없는 메소드이다. 따라서 이를 상속받은 자식 클래스는 반드시 실체적인 실행 내용을 오버라이딩(재정의)해야 한다.
*   **➍ 자식의 추상화**: 부모로부터 물려받은 추상 메소드를 자식 클래스에서 구현(재정의)하지 않으면, 자식 클래스도 여전히 미완성된 부분을 포함하게 된다. 이 경우 자식 클래스 역시 `abstract`를 붙여 추상 클래스로 선언해야 컴파일 에러가 발생하지 않는다.

## 6. 부모 생성자 호출 에러 해결
**Q. Child 생성자에서 컴파일 에러가 발생한 이유와 해결 방법을 설명해보세요.**
![img.png](img.png)
### **❌ 에러 발생 이유**
자식 객체를 생성할 때는 부모 객체가 먼저 생성되어야 하며, 자식 생성자 첫 줄에는 반드시 부모 생성자를 호출하는 `super()`가 있어야 한다.

1.  개발자가 자식 생성자에 `super()`를 직접 작성하지 않으면 컴파일러가 자동으로 `super();`(기본 생성자 호출)를 삽입한다.
2.  하지만 부모 클래스인 `Parent`에는 `Parent(String name)`라는 **사용자 정의 생성자**만 있고, 매개변수가 없는 **기본 생성자가 없다**.
3.  결과적으로 존재하지 않는 부모의 기본 생성자를 호출하려다 보니 컴파일 에러가 발생한다.

### **✅ 해결 방법**
자식 생성자 첫 줄에 `super(name);`을 추가하여, 부모의 사용자 정의 생성자를 명시적으로 호출해야 한다.
```java
public class Child extends Parent {
    public int studentNo;

    public Child(String name, int studentNo) {
        super(name); // 해결 방법: 부모 생성자를 명시적으로 호출한다.
        this.name = name;
        this.studentNo = studentNo;
    }
}
```

    *   호출했던 **`Parent()`로 돌아와** 남은 코드 실행 → `"Parent() call"` 출력
    *   호출했던 **`Child(String name)`로 돌아와** 남은 코드 실행 → `"Child(String name) call"` 출력
    *   마지막으로 **`Child()`로 돌아와** 남은 코드 실행 → `"Child() call"` 출력

## 9. 자동 타입 변환과 상속 관계

    *   `B`는 `A`의 자식이다.
    *   `D`, `E`는 `B`의 자식이다.
    *   결국 `B` 타입의 변수나 매개변수에는 **B 본인**과 **B를 상속받은 하위 객체(D, E)**만 들어갈 수 있다.

2.  **보기별 분석**
    *   ➊ `new B()`: 본인 타입이므로 대입 가능하다.
    *   ➌ `new D()`, ➍ `new E()`: `B`의 자식 객체이므로 `B` 타입으로 **자동 타입 변환**되어 대입 및 메소드 호출이 가능하다.
    *   ➋ `(B) new A()`: **정답이다.** 부모 객체인 `A`를 강제로 자식 타입인 `B`로 변환하려고 한다. 강제 타입 변환은 **이미 자식 객체가 부모 타입으로 변환되어 있는 상태**에서만 가능하다. 처음부터 부모(`A`)로 생성된 객체는 자식 타입으로 바꿀 수 없으며 실행 시 `ClassCastException`이 발생한다.

## 10. 추상 메소드와 재정의 의무
**Q. 다음과 같이 작성한 Computer 클래스에서 컴파일 에러가 발생했습니다. 그 이유를 설명해보세요.**

```java
public abstract class Machine {
    public void powerOn() { }
    public void powerOff() { }
    public abstract void work();
}

public class Computer extends Machine {
}
``` 
### ❌ 에러 발생 이유

> 부모 클래스인 Machine이 추상 클래스이며, 그 안에 실행 내용이 없는 추상 메소드(public abstract void work();)를 포함하고 있기 때문이다.

> 자식의 구현 의무: 추상 클래스를 상속받는 일반(실체) 클래스는 부모 클래스에 선언된 모든 추상 메소드를 반드시 재정의(Overriding)하여 실행 내용을 작성해야 한다.

> 구현 누락: 현재 Computer 클래스는 Machine을 상속받았지만, 추상 메소드인 work()를 재정의하지 않았다.

> 결과: 미완성된 메소드(work)를 그대로 가지고 있는 Computer 클래스 역시 미완성 상태가 된다. 자바에서 미완성된 클래스는 일반 클래스로 선언할 수 없으므로 컴파일 에러가 발생한다.

### 💡 해설

부모 클래스인 `Machine`이 **추상 클래스**이며, 그 안에 실행 내용이 없는 **추상 메소드**(`public abstract void work();`)를 포함하고 있기 때문이다.

1.  **자식의 구현 의무**: 추상 클래스를 상속받는 일반(실체) 클래스는 부모 클래스에 선언된 모든 추상 메소드를 반드시 **재정의(Overriding)**하여 실행 내용을 작성해야 한다.
2.  **구현 누락**: 현재 `Computer` 클래스는 `Machine`을 상속받았지만, 추상 메소드인 `work()`를 재정의하지 않았다.
3.  **결과**: 미완성된 메소드(`work`)를 그대로 가지고 있는 `Computer` 클래스 역시 미완성 상태가 된다. 자바에서 미완성된 클래스는 일반 클래스로 선언할 수 없으므로 컴파일 에러가 발생한다.

---

### ✅ 해결 방법

**방법 1: 추상 메소드 재정의 (가장 권장됨)**
`Computer` 클래스 내부에 부모의 추상 메소드를 구체적으로 구현한다.
```java
public class Computer extends Machine {
    @Override
    public void work() {
        System.out.println("컴퓨터가 프로그램을 실행하며 작업을 수행한다.");
    }
}
```

## 12. instanceof 연산자와 강제 타입 변환
**Q. action() 메소드 호출 시 매개값이 C 객체일 경우에만 method2()가 호출되도록 밑줄에 들어갈 코드를 작성한다.**
![img_5.png](img_5.png)
![img_6.png](img_6.png)
### ✅ 정답
```java
a instanceof C c
``` 

### 💡 해설

매개변수로 부모 타입인 `A`를 받고 있기 때문에, 전달된 객체가 실제로 `C`인지 확인하는 과정이 필요하다.

1.  **instanceof 연산자**: 객체가 특정 클래스의 인스턴스인지 확인하는 연산자다. `a instanceof C`는 "변수 `a`가 참조하는 실제 객체가 `C` 타입인가?"를 확인한다.
2.  **형변환의 필요성**: 부모 타입인 `A` 변수로는 자식 클래스인 `C`에만 있는 고유한 메소드인 `method2()`에 접근할 수 없다. 따라서 타입을 `C`로 변환하는 과정이 필요하다.
3.  **패턴 매칭 (Java 12 이상)**: `if (a instanceof C c)` 구문을 사용하면 타입을 확인하는 동시에 변수 `c`에 `C` 타입으로 형변환된 객체를 바로 대입한다. 덕분에 코드 블록 안에서 별도의 강제 타입 변환(`(C) a`) 없이 `c.method2()`를 바로 호출할 수 있다.

### 📌 핵심 

* **용도**: 부모 타입 매개변수를 가진 메소드 내에서, 특정 자식 클래스에만 존재하는 고유 기능을 사용해야 할 때 사용한다.
* **안전성**: `instanceof` 없이 강제 형변환을 시도하면 다른 객체(`A`나 `B`)가 들어왔을 때 `ClassCastException` 에러가 발생하므로, 반드시 이 연산자로 타입을 확인해야 한다.
* **작동 원리**: `a.method1()`은 다형성에 의해 각 객체의 재정의된 내용이 알아서 실행되지만, 자식의 개별 메소드는 이처럼 타입을 체크하고 정밀하게 타격하여 호출한다.
