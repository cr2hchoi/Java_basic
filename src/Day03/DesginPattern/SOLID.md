# 🏛️ 객체 지향 설계의 5대 원칙: SOLID

SOLID는 소프트웨어를 더 이해하기 쉽고, 유연하며, 유지보수하기 쉽게 만들기 위한 다섯 가지 원칙의 앞글자를 딴 핵심 가이드라인임.

---

## 1. SRP: 단일 책임 원칙 (Single Responsibility Principle)
> **"클래스는 단 하나의 책임(기능)만 가져야 한다."**

- **핵심**: 클래스를 변경해야 하는 이유는 오직 하나여야 함.
- **장점**: 코드의 가독성이 좋아지고, 특정 기능 수정 시 다른 기능에 미치는 영향이 최소화됨.
- **사례**: 유저 정보 저장 로직과 이메일 발송 로직은 서로 다른 클래스로 분리해야 함.

---

## 2. OCP: 개방-폐쇄 원칙 (Open-Closed Principle)
> **"확장에는 열려 있고, 수정에는 닫혀 있어야 한다."**

- **핵심**: 기존의 코드를 변경하지 않으면서 기능을 추가할 수 있도록 설계해야 함.
- **방법**: **인터페이스**와 **다형성**을 활용해 변하는 부분만 추상화함.
- **사례**: 새로운 결제 수단(카카오페이)이 추가될 때, 기존의 결제 로직 코드를 수정하는 대신 결제 인터페이스를 상속받는 새 클래스를 추가함.



---

## 3. LSP: 리스코프 치환 원칙 (Liskov Substitution Principle)
> **"자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다."**

- **핵심**: 부모 클래스의 인스턴스 대신 자식 클래스의 인스턴스를 사용해도 프로그램의 의미가 변하지 않고 정상적으로 동작해야 함.
- **장점**: 다형성을 안전하게 보장함.
- **사례**: '직사각형' 클래스를 상속받은 '정사각형' 클래스가 부모의 가로/세로 설정 규칙을 깨뜨린다면 LSP 위반임.



---

## 4. ISP: 인터페이스 분리 원칙 (Interface Segregation Principle)
> **"사용하지 않는 메서드에 의존하도록 강제해서는 안 된다."**

- **핵심**: 거대한 인터페이스 하나보다, 구체적인 여러 개의 인터페이스가 낫다.
- **방법**: 인터페이스를 사용하는 클라이언트의 목적에 맞게 세분화하여 쪼개야 함.
- **사례**: 스마트폰 인터페이스에 '전화', '문자', '무선충전' 기능이 뭉쳐있을 때, 무선충전이 안 되는 구형 폰을 위해 '무선충전' 인터페이스만 따로 분리함.

---

## 5. DIP: 의존역전 원칙 (Dependency Inversion Principle)
> **"상위 모듈은 하위 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 한다."**

- **핵심**: 구체적인 클래스(구현체)보다 인터페이스나 추상 클래스에 의존해야 함.
- **방법**: 의존성 주입(DI, Dependency Injection)을 통해 변화에 유연하게 대응함.
- **사례**: 자동차(상위)가 한국타이어(하위)라는 구체적인 모델에 의존하는 것이 아니라, '타이어'라는 인터페이스에 의존하게 설계함.



---

## 💡 SOLID 요약 한 줄 팁
1. **SRP**: 섞지 마라.
2. **OCP**: 건드리지 말고 추가해라.
3. **LSP**: 부모님 이름에 먹칠하지 마라(상속 규칙 준수).
4. **ISP**: 필요한 것만 골라라.
5. **DIP**: 구체적인 것에 집착하지 말고 추상적인 틀에 의존해라.
